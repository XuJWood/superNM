# Init Project — 新项目全流程初始化

从产品调研到持续开发的完整项目启动流程。可移植：拷贝此文件夹到任意项目的 `.claude/skills/` 下即可使用。

## 前置检查

启动前确认以下 skills/plugins 已安装（未安装则提示用户安装）：

**必装：**
- `frontend-design` — 高质量前端界面生成（检查 `~/.claude/settings.json` 的 `enabledPlugins`）
- `superpowers` — 多 agent 协作增强（检查 `~/.claude/plugins/cache/claude-plugins-official/superpowers/`）

**推荐安装命令（如未装）：**
告知用户在 Claude Code 中运行：
```
/install-plugin frontend-design
/install-plugin superpowers
```

---

## 执行流程

用户调用: `/superNM $ARGUMENTS`

如果 $ARGUMENTS 有值，作为项目名称。否则进入交互式流程。

---

### Phase 1: 产品方向收集

**向用户询问以下信息（必须全部收集）：**

1. **产品方向** — 你要做什么产品？（接受：文字描述 / 竞品链接 / 参考文档路径 / 截图）
2. **目标用户** — 给谁用的？
3. **核心功能** — 最重要的 3-5 个功能点
4. **技术偏好** — 后端语言/框架偏好（根据产品方向推荐，不预设特定架构）
5. **是否需要前端** — 需要哪些前端界面？（管理后台/用户端/监控面板/都要）
6. **部署环境** — 本地开发 / 云服务器 / Docker / K8s

如果用户给了链接，用 WebFetch 抓取内容分析。如果给了参考项目路径，用 Read 读取。

---

### Phase 2: 产品调研 + 技术文档

基于 Phase 1 收集的信息，**进行产品调研**：

1. 用 WebSearch 搜索：

   - 开源方案参考
   - 行业最佳实践

2. 生成 `docs/Product-Report.md`，结构如下：

|------|----------|--------|------|------|

## 三、技术方案

- 后端框架选择及理由
- 数据库选型

- 难点 1 — 解决思路

| 层 | 技术 | 理由 |

### Phase 3: 生成开发计划 (ROADMAP.md)

基于 Product-Report.md，生成 `ROADMAP.md`：

**关键规则：**
- 用 **阶段 (Phase)** 划分，**不要用时间/周次**
- 每个阶段有明确的交付物和验收标准
- 标注阶段间依赖关系

```markdown
# {项目名} 开发计划

> 按阶段划分。每次开发前读取，完成后更新状态。

---

## 当前阶段: Phase 0 🔲
## 下一阶段: Phase 1

---

### Phase 0 — 基础设施 🔲

**目标：** {一句话}
**交付物：** {列出}
**验收标准：** {可度量}

| 状态 | 功能点 | 开发内容 | 验收标准 |
|------|--------|----------|----------|
| 🔲 | ... | ... | ... |

---

### Phase 1 — 核心功能 🔲

**目标：** ...
**依赖：** Phase 0 完成
**交付物：** ...

| 状态 | 功能点 | 开发内容 | 验收标准 |
|------|--------|----------|----------|
| 🔲 | ... | ... | ... |

---

### Phase 2 — 前端界面 🔲
（如有前端需求）

### Phase N — 生产强化 🔲
...
```

### Phase 4: 前端界面规划

**向用户确认前端需求：**

1. 需要哪些页面/界面？
2. 每个页面的核心功能是什么？
3. 设计风格偏好？（深色/浅色/现代/极简/科技感）
4. 是否有参考设计？（链接/截图）

收集后：
- 更新 ROADMAP.md，将前端开发插入对应阶段（通常与后端功能同阶段）
- 记录到 `docs/Frontend-Design-Spec.md`

```markdown
# 前端界面设计规格

## 页面清单
| 页面 | 功能 | 所属阶段 | 技术方案 |
|------|------|----------|----------|
| ... | ... | Phase X | HTML/React/Vue |

## 设计规范
- 主题: 深色/浅色
- 配色: ...
- 字体: ...
- 响应式: 是/否

## 各页面详细设计
### 页面 1: {名称}
- 功能描述
- 核心组件
- 交互逻辑
```

**开发前端时调用 `/frontend-design` skill 生成高质量界面代码。**

### Phase 5: 生成项目骨架

**根据 Phase 2 (Product-Report) 的技术选型和 Phase 3 (ROADMAP) 的功能模块，动态生成项目结构。**

核心原则：**架构由调研和计划驱动，不预设特定技术栈。**

生成流程：
1. 依据 Product-Report 确认的技术栈，确定包管理工具和项目配置文件
2. 依据 ROADMAP 的功能模块，确定源码目录结构
3. 创建基础目录和核心入口文件
4. 生成 `.gitignore` 和 `.claude/settings.json`

```bash
# 通用基础结构（根据实际技术栈和模块灵活调整）
mkdir -p src          # 源码目录（按模块划分子目录）
mkdir -p config       # 配置文件目录
mkdir -p docs         # 文档目录
mkdir -p tests        # 测试目录
mkdir -p reports      # 周报/日报目录
mkdir -p scripts      # 脚本目录
```

生成的核心文件（按需，参考 ROADMAP 模块）：
1. 项目配置文件（`pyproject.toml` / `package.json` / `go.mod` — 取决于技术栈）
2. 主入口文件（`main.py` / `index.ts` / `main.go` — 取决于语言）
3. 配置管理模块
4. `.gitignore`
5. `.claude/settings.json`
6. Git 初始化

### Phase 6: 生成开发规范 (CLAUDE.md)

生成项目的 CLAUDE.md，包含：

```markdown
# CLAUDE.md

> **开发基线文件** — 每次会话自动加载。

## 开发工作流（每次会话必须遵循）
① 读取本文件 → ② 读取 ROADMAP.md → ③ 读取 devlog.md（最近一条）→ ④ 开始开发 → ⑤ 完成后更新

## 项目简介
{基于 Product-Report.md}

## 架构核心
{基于技术选型}

## 开发规范
- 新增模块的标准流程
- 命名规范
- 禁止事项

## 常用命令
{启动/测试/依赖管理}

## 目录结构
{实际目录树}

## 关键文件索引
| 文件 | 用途 | 何时读取 |
|------|------|---------|
| CLAUDE.md | 架构规范 | 每次会话开始 |
| ROADMAP.md | 开发计划 | 每次会话开始 |
| devlog.md | 开发日志 | 每次会话开始 |
| docs/Product-Report.md | 产品调研 | 了解产品背景 |

## 开发完成流程（每次必须执行）
1. 更新 devlog.md（顶部追加）
2. 更新 ROADMAP.md（勾选完成项）
3. git add + git commit
4. 验证服务可用
```

### Phase 7: 初始化开发日志 + 日报/周报体系

#### 7.1 文件存放规范（强制）

```
项目根目录/
├── devlog.md                       # 开发日志（每次开发追加，颗粒度=单次开发任务）
└── reports/
    ├── _模板.md                    # 周报模板（生成新周报时复制）
    ├── _日报模板.md                # 日报模板
    ├── daily/                      # 日报存放目录
    │   └── YYYY-MM-DD.md           # 命名规则：日期.md
    └── week-NN-MMDD.md             # 周报：week-周次-起始日期.md（如 week-02-0601.md）
```

**命名规则（严格遵守）：**
- 日报: `reports/daily/2026-06-04.md`（当天日期）
- 周报: `reports/week-NN-MMDD.md`（NN=两位周次, MMDD=本周一日期, 例: `week-03-0608.md`）
- 开发日志: `devlog.md`（项目根目录, 单文件持续追加）

#### 7.2 触发时机规范（强制）

| 文档 | 触发时机 | 由谁触发 |
|------|---------|---------|
| **devlog.md** | 每完成一个开发任务（功能/修复/重构）立即追加 | 开发会话结束前必做 |
| **日报** | **每天最后一次开发会话结束时生成**（汇总当天 devlog 新增） | 当天工作结束前 |
| **周报** | **每周日 18:00 后**或**每周一开工前**生成（汇总本周日报） | 周末/周初首次会话 |

**会话开始检查清单：**
1. 读取 CLAUDE.md → ROADMAP.md → devlog.md
2. 检查今天日期：
   - 如果是周一且 `reports/` 下没有上周的周报 → 提示用户生成上周周报
   - 如果 `reports/daily/` 下没有昨天的日报且昨天有 devlog 更新 → 提示用户补昨日日报
3. 才开始当前任务

#### 7.3 模板内容

**devlog.md 初始化：**
```markdown
# {项目名} 开发日志

> 每次开发完成后在顶部追加。格式：日期 + 完成项 + 验证结果 + 下一步。
> 颗粒度：单次开发任务（一个功能/一个修复/一次重构）。

---
```

**reports/_日报模板.md：**
```markdown
# {项目名} 日报 — YYYY-MM-DD

**日期：** YYYY-MM-DD（周X）
**所属阶段：** Phase N
**当日工时：** X 小时

---

## 一、今日完成
1. 完成项 — 简要说明（关联 devlog 时间戳）
2. ...

## 二、今日进展
| 任务 | 所属阶段 | 状态 | 备注 |
|------|---------|------|------|
| ... | Phase N | ✅/🔲 | ... |

## 三、遇到的问题
-

## 四、明日计划
1.
```

**reports/_模板.md（周报）：**
```markdown
# {项目名} 周报 — Week N

**日期：** YYYY-MM-DD ~ YYYY-MM-DD（周X ~ 周X）
**项目整体进度：** ≈ X%
**当前阶段：** Phase N

---

## 一、工作详情

> 按 **阶段/模块** 组织，**不按日期分小节**；每块标题带阶段进度 %。

**阶段/模块名称（阶段进度 X%）**

1. 完成项 — 具体说明（做了什么 + 效果/数据）
2. ...

**另一阶段/模块（阶段进度 X%）**

1. ...

---

## 二、工作汇总

### 任务情况
| 序号 | 任务项 | 所属阶段 | 完成状态 |
|------|--------|----------|----------|
| 1 | ... | Phase N | ✅ 已完成 |
| 2 | ... | Phase N | 🔲 进行中 |

### 任务进展

> 按 **类别** 分组（如 核心管道 / 稳定性攻坚 / 测试对比），组内连续编号，每条用「完成了XXX — 具体说明 + 效果」格式。

**类别一**

1. 完成了XXX — 具体实现 + 达成效果
2. ...

**类别二**

3. 完成了XXX — ...

---

## 待解决问题
-

## 下周计划
1.
```

### Phase 9: 输出总结

向用户汇报：
1. 项目结构树
2. 生成的核心文件清单
3. 下一步建议（从 ROADMAP.md Phase 0 开始）
4. 提醒：开发时可用 `/frontend-design` 生成前端界面

## 持续开发规范

项目初始化后，后续每次开发会话遵循 CLAUDE.md 中的工作流：
1. 读取 CLAUDE.md + ROADMAP.md + devlog.md
2. 从当前阶段的下一个 🔲 任务开始
3. 开发完成后更新 devlog + ROADMAP + commit
4. 每周末生成周报到 reports/

## 可移植性

将 `.claude/skills/supernm/` 整个文件夹拷贝到新项目即可：
```bash
cp -r .claude/skills/supernm /path/to/new-project/.claude/skills/
```

或放到用户级 `~/.claude/commands/` 全局可用。
